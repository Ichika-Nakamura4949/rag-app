# RAGパイプラインの処理フロー

## 全体像

```
ユーザーがPDFをアップロード
  │
  ▼
┌─────────────────────────────────────────────────┐
│  ingestion.py  （① PDF解析 → ② チャンク化 → ③ 埋め込み → ④ DB保存）│
└─────────────────────────────────────────────────┘
  │
  │  保存完了
  ▼
┌──────────────┐
│  ChromaDB    │ ← ベクトルデータがここに溜まる
└──────────────┘
  │
  │  ユーザーが質問する
  ▼
┌──────────────────────────────────────────────────┐
│  chain.py  （⑤ 質問を英語に翻訳）                    │
│      ↓                                            │
│  retriever.py  （⑥ ChromaDBからベクトル検索）        │
│      ↓                                            │
│  chain.py  （⑦ 検索結果 + 質問 → GPT-4o → ⑧ 回答）  │
└──────────────────────────────────────────────────┘
```

設定値（チャンクサイズ、検索件数など）は config.py で管理。

---

## 各ステップの詳細

### ① PDF解析（ingestion.py `_parse_pdf`）

PDFファイルを開いて、全ページのテキストを取り出し、1つの長い文字列にまとめる。

- PyMuPDFというライブラリを使用
- 1ページずつテキストを取得し、空白だけのページは飛ばす
- 全ページを `\n\n` で結合

例：10ページの論文 → `"1ページ目のテキスト\n\n2ページ目のテキスト\n\n..."` という1つの文字列になる。

---

### ② チャンク化（ingestion.py `_splitter.split_text`）

①で作った長い文字列を800文字ずつに切り分ける。隣同士のチャンクで200文字が重なるようにする。

- 設定: `chunk_size=800`、`chunk_overlap=200`（config.pyで定義）
- LangChainの `RecursiveCharacterTextSplitter` を使用

なぜ重ねるのか: 文の途中でバッサリ切ると意味がわからなくなるので、少し重複を持たせて文脈を保つ。

```
元のテキスト: [============================]（全体で2000文字とする）

チャンク1: [========]           ← 0〜800文字目
チャンク2:       [========]     ← 600〜1400文字目（200文字重なる）
チャンク3:             [========] ← 1200〜2000文字目（200文字重なる）
```

---

### ③ 埋め込み（ingestion.py `vectorstore.add_documents` 内部で自動実行）

各チャンクのテキストをOpenAI Embedding APIに送信して、1536個の数値の列（ベクトル）に変換する。

- 使用モデル: `text-embedding-3-small`（config.pyで定義）
- 意味が似たテキストは似た数値になる性質を利用して、後で検索する

```
「犬が走る」  → [0.12, 0.85, 0.33, ...]  ← 似た数値
「猫が走る」  → [0.11, 0.82, 0.35, ...]  ← 似た数値
「経済の動向」→ [0.91, 0.04, 0.67, ...]  ← 全然違う数値
```

---

### ④ DB保存（ingestion.py `vectorstore.add_documents` 内部で自動実行）

③の埋め込みと同時に、以下のセットをChromaDBに保存する。

- テキスト（チャンクの中身）
- ベクトル（数値の列）
- メタデータ（ファイル名、ドキュメントID、チャンク番号）

ChromaDBはローカルの `./data/chroma` フォルダに永続化される。

---

### ⑤ 質問の英語翻訳（chain.py `_translate_query`）

ユーザーの日本語の質問をGPT-4oで英語に翻訳する。英語のPDFを英語で検索するための前処理。

```
入力:「データ収集に使われたロボットのハードウェア構成は？」
  ↓ GPT-4oが翻訳
出力: "What is the hardware configuration of the robot used for data collection?"
```

---

### ⑥ ベクトル検索（retriever.py `get_retriever` → chain.pyで `retriever.invoke`）

翻訳された英語の質問をEmbedding APIで数値に変換し、ChromaDBの中の全チャンクの数値と比較して、最も近い上位8件を取得する。

- 設定: `retriever_k=8`（config.pyで定義）
- 「数値が近い」＝「意味が似ている」と判断される

```
翻訳された質問 "What is the hardware configuration..."
    ↓ 数値に変換
[0.45, 0.72, 0.11, ...]
    ↓ ChromaDBの全チャンクと比較
    ↓ 上位8件を取得
[チャンクA, チャンクB, ... チャンクH]
```

---

### ⑦⑧ 回答生成（chain.py `invoke`）

⑥で取得した8件のチャンクのテキストと、元の日本語の質問をGPT-4oに渡す。GPT-4oはチャンクの中から答えを探して日本語で回答する。

GPT-4oに渡される内容のイメージ:

```
【システム】
あなたは社内ドキュメントに基づいて質問に回答するアシスタントです。
コンテキスト:
[チャンクAのテキスト]
---
[チャンクBのテキスト]
---
... (8件分)

【ユーザー】
データ収集に使われたロボットのハードウェア構成と、データ収集に要した期間を教えてください。
```

コンテキストに答えが含まれていれば回答し、含まれていなければ「該当する情報が見つかりませんでした」と返す。

## ファイルと処理の対応表

| ステップ | 処理 | ファイル |
|---|---|---|
| ① | PDF → テキスト抽出 | ingestion.py |
| ② | テキスト → 800文字ずつに分割 | ingestion.py |
| ③ | チャンク → 数値に変換（埋め込み） | ingestion.py（内部で自動） |
| ④ | 数値 + テキスト → ChromaDBに保存 | ingestion.py（内部で自動） |
| ⑤ | 日本語の質問 → 英語に翻訳 | chain.py |
| ⑥ | 英語の質問で似たチャンクを8件検索 | retriever.py |
| ⑦ | 検索結果 + 質問をGPT-4oに渡す | chain.py |
| ⑧ | GPT-4oが日本語で回答を返す | chain.py |

## 設定値一覧（config.py）

| 設定 | 値 | 説明 |
|---|---|---|
| `chunk_size` | 800 | チャンクの最大文字数 |
| `chunk_overlap` | 200 | 隣接チャンクの重複文字数 |
| `retriever_k` | 8 | 検索で取得するチャンク数 |
| `embedding_model` | text-embedding-3-small | 埋め込みに使うOpenAIモデル |
| `llm_model` | gpt-4o | 回答生成に使うOpenAIモデル |
| `temperature` | 0 | 回答のランダム性（0=最も確定的） |

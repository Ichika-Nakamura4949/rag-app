## RAGパイプライン

### Ingestion Pipeline（ドキュメント取り込み）

ユーザーがドキュメントをアップロードしてからベクトルDBに保存されるまでのフロー。

```
PDF/DOCX Upload
      │
      ▼
┌─────────────┐
│  ファイル    │  PyPDF2 / python-docx / Unstructured
│  パース     │  → テキスト抽出
└──────┬──────┘
       │
       ▼
┌─────────────┐
│  チャンク   │  RecursiveCharacterTextSplitter
│  分割       │  chunk_size=1000, overlap=200
└──────┬──────┘
       │
       ▼
┌─────────────┐
│  Embedding  │  OpenAI text-embedding-3-small
│  生成       │  → 1536次元ベクトル
└──────┬──────┘
       │
       ▼
┌─────────────┐
│  ベクトル   │  ChromaDB
│  DB保存     │  メタデータ（ファイル名、ページ番号等）付き
└─────────────┘
```

### Query Pipeline（質問応答）

ユーザーの質問に対してRAGで回答を生成するフロー。

```
ユーザー質問
      │
      ▼
┌─────────────┐
│  Query      │  OpenAI text-embedding-3-small
│  Embedding  │  → 質問をベクトル化
└──────┬──────┘
       │
       ▼
┌─────────────┐
│  類似検索   │  ChromaDB similarity search
│  (Top-K)    │  k=4（関連チャンクを4件取得）
└──────┬──────┘
       │
       ▼
┌─────────────┐
│  プロンプト │  システムプロンプト + 検索結果 + 質問
│  構築       │  → コンテキスト付きプロンプト
└──────┬──────┘
       │
       ▼
┌─────────────┐
│  LLM生成    │  GPT-4o
│             │  → 検索結果に基づく回答
└──────┬──────┘
       │
       ▼
回答 + 参照元情報
```

## 技術選定と理由

### バックエンド: FastAPI

- 非同期処理のネイティブサポート（ファイルアップロード、API呼び出し）
- 自動APIドキュメント生成（Swagger UI）
- Pydanticによる型安全なリクエスト/レスポンス管理
- Pythonエコシステムとの親和性（LangChain、ChromaDB等）

### RAGフレームワーク: LangChain

- ドキュメントローダー、テキスト分割、ベクトルストアの統合抽象化
- RAGチェーンの構築が容易
- 豊富なドキュメントと活発なコミュニティ

### ベクトルDB: ChromaDB

- ローカル実行可能でインフラ不要
- LangChainとの統合が容易
- 開発・学習用途に最適（本番ではPinecone/Qdrant等へ移行可能）

### Embedding: OpenAI text-embedding-3-small

- コスト効率が良い（text-embedding-3-largeより安価）
- 十分な精度（社内ドキュメントの検索用途に適切）
- OpenAI APIで統一的に管理可能

### LLM: GPT-4o

- 高精度な日本語回答生成
- コンテキストウィンドウが大きい（128K tokens）
- コストと性能のバランスが良い

### フロントエンド: Next.js + Tailwind CSS

- App Routerによるモダンなルーティング
- Server Componentsによるパフォーマンス最適化
- Tailwind CSSによる高速なUI開発

## ディレクトリ構成の設計方針

```
backend/app/
├── main.py              # FastAPIアプリケーション初期化、ルーター登録
├── api/
│   ├── routes/
│   │   ├── chat.py      # POST /api/chat — 質問送信・回答取得
│   │   └── documents.py # POST /api/documents — アップロード
│   │                     # GET  /api/documents — 一覧取得
│   │                     # DELETE /api/documents/{id} — 削除
│   └── deps.py          # 共通の依存性注入（DB接続、設定等）
├── core/
│   ├── config.py        # 環境変数、アプリケーション設定
│   └── rag/
│       ├── ingestion.py # ドキュメント解析 → チャンク分割 → Embedding → 保存
│       ├── retriever.py # ベクトル検索、関連チャンク取得
│       └── chain.py     # LangChain RAGチェーン構築・実行
├── models/              # Pydanticモデル（リクエスト/レスポンス）
└── services/            # ビジネスロジック（ファイル管理等）
```

- **api/routes**: HTTPエンドポイントの定義。薄く保ち、ロジックはservicesやcoreに委譲
- **core/rag**: RAGパイプラインの中核。Ingestion/Query両パイプラインを実装
- **models**: リクエスト/レスポンスのPydanticスキーマ
- **services**: ファイル保存・削除などのビジネスロジック
